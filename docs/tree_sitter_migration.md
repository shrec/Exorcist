# Tree-sitter Migration Plan

## Current State

Exorcist uses a custom `QSyntaxHighlighter` subclass (`SyntaxHighlighter`) with per-line regex rules. Each language is defined by a builder function in `src/editor/languages/lang_*.cpp` that populates a `QVector<Rule>` with `QRegularExpression` patterns plus `QTextCharFormat` styles. There are 13 language files covering 60+ syntaxes.

### Strengths
- Zero dependencies — pure Qt, no external library
- Fast for small files — regex compilation is once per highlighter instance
- Shebang detection and filename-based fallback already implemented
- Clean builder function interface (`LangBuildFn`) makes adding languages trivial

### Limitations
- **Per-line processing**: `highlightBlock()` operates one line at a time. Multi-line constructs (string interpolation, heredocs, nested comments) require fragile block-state tracking.
- **No parse tree**: Regex patterns cannot distinguish semantic roles (variable vs function call vs type name with the same identifier).
- **No incremental reparsing**: Editing line N re-highlights the entire visible range; there is no way to skip unchanged regions.
- **No error recovery**: A malformed string literal breaks highlighting for the rest of the block.

## Why Tree-sitter

Tree-sitter produces a concrete syntax tree (CST) from incremental parsing with error recovery. It handles:
- Nested constructs (template strings, block comments inside strings, etc.)
- Semantic node types (`function_definition`, `type_identifier`, `variable_declarator`)
- Incremental edits — only the changed subtree is reparsed
- Consistent grammars maintained by the community for 100+ languages

## Integration Strategy

### Step 1: CMake ExternalProject

Add tree-sitter as a Git submodule or `FetchContent` dependency:

```cmake
include(FetchContent)

FetchContent_Declare(
    tree-sitter
    GIT_REPOSITORY https://github.com/tree-sitter/tree-sitter.git
    GIT_TAG        v0.25.3
)
FetchContent_MakeAvailable(tree-sitter)
```

Tree-sitter's core is a single C library (~15 source files, MIT license). Build it as a static library linked into the Exorcist executable. No DLL distribution needed.

### Step 2: Grammar Management

Each language grammar is a separate C file generated by tree-sitter CLI from a `grammar.js`. For the prototype, bundle grammars for the most-used languages:

| Priority | Grammar | NPM package |
|----------|---------|-------------|
| 1 | C / C++ | tree-sitter-c, tree-sitter-cpp |
| 2 | Python | tree-sitter-python |
| 3 | JavaScript / TypeScript | tree-sitter-javascript, tree-sitter-typescript |
| 4 | Rust | tree-sitter-rust |
| 5 | Go | tree-sitter-go |
| 6 | JSON / YAML / TOML | tree-sitter-json, tree-sitter-yaml, tree-sitter-toml |

Each grammar adds two C files (`parser.c` + optional `scanner.c`), totaling ~50-200 KB per language. They compile to static objects — no runtime codegen.

Store grammars under `src/editor/grammars/` with one directory per language:
```
src/editor/grammars/
├── CMakeLists.txt          # builds all grammar static libs
├── cpp/parser.c
├── cpp/scanner.c
├── python/parser.c
├── python/scanner.c
├── javascript/parser.c
├── javascript/scanner.c
└── ...
```

### Step 3: Highlighting Bridge

Create a new class `TreeSitterHighlighter` alongside the existing `SyntaxHighlighter`:

```
src/editor/
├── syntaxhighlighter.h/cpp      # existing regex highlighter (kept as fallback)
├── treesitterhighlighter.h/cpp   # new tree-sitter highlighter
├── highlighterfactory.h          # unified factory: prefers tree-sitter, falls back to regex
```

**Key types:**

```cpp
class TreeSitterHighlighter : public QSyntaxHighlighter {
public:
    TreeSitterHighlighter(const TSLanguage *lang, QTextDocument *doc);

    // Called by Qt's text framework on each edit
    void highlightBlock(const QString &text) override;

    // Called when document text changes — feeds edit to tree-sitter
    void onContentsChange(int position, int charsRemoved, int charsAdded);

private:
    TSParser *m_parser = nullptr;
    TSTree   *m_tree   = nullptr;
    QByteArray m_sourceUtf8;           // full document text in UTF-8
    std::unordered_map<TSSymbol, QTextCharFormat> m_formatMap;
};
```

**Workflow:**
1. On first open: parse full document → build `TSTree`
2. On each edit (`contentsChange` signal): compute `TSInputEdit`, call `ts_parser_parse()` incrementally
3. On `highlightBlock()`: walk the tree nodes intersecting the current block's range, apply `QTextCharFormat` from `m_formatMap` keyed by node type

### Step 4: Node-type → Format Mapping

Map tree-sitter node types to Exorcist's existing colour palette (`langcommon.h`):

| Node type pattern | Format |
|-------------------|--------|
| `string_literal`, `string`, `template_string` | `clr::string_` |
| `comment`, `line_comment`, `block_comment` | `clr::comment_` |
| `number_literal`, `integer_literal`, `float_literal` | `clr::number_` |
| `function_declarator`, `function_definition` → name child | `clr::function_` |
| `type_identifier`, `primitive_type`, `builtin_type` | `clr::type_` |
| `preproc_*` | `clr::preproc_` |
| `true`, `false`, `null`, `nullptr` | `clr::literal_` |
| Keywords (matched via `ts_node_is_named` + symbol list) | `clr::keyword` |

This mapping lives in a data table, not in code, so adding a new language requires only adding the grammar C files.

### Step 5: Background Parsing (Phase 4 optimization)

For large files (>10K lines), move parsing off the UI thread:

```
┌──────────────┐         ┌──────────────────┐
│  UI Thread   │         │  Worker Thread   │
│              │  edit   │                  │
│  EditorView ─┼────────►│  TSParser::parse │
│              │         │         │        │
│  apply fmt  ◄┼─────────┤  emit ranges     │
│              │  signal │                  │
└──────────────┘         └──────────────────┘
```

Use Qt's `QThreadPool` with a single-shot `QRunnable`. Emit a signal with the list of changed ranges + their formats. The UI thread applies formats via `QSyntaxHighlighter::rehighlightBlock()`.

### Step 6: LSP Semantic Tokens Overlay

Once tree-sitter provides syntactic highlighting, layer LSP `textDocument/semanticTokens/full` on top:

1. Request semantic tokens from the language server
2. Parse the delta-encoded response into `(line, col, length, tokenType, modifiers)`
3. Override tree-sitter formats where LSP provides finer-grained information (e.g., distinguishing local variables from parameters, or marking deprecated symbols with strikethrough)

This gives two levels of quality:
- **Instant**: tree-sitter highlighting (local, <1ms for typical edits)
- **Rich**: LSP semantic tokens (remote, arrives after 50-200ms server response)

## Migration Path

| Phase | Scope | Deliverable |
|-------|-------|-------------|
| **A** | Add tree-sitter core via FetchContent | `tree-sitter` static lib in CMake |
| **B** | Bundle C++ grammar, implement `TreeSitterHighlighter` | C++ files highlighted by tree-sitter |
| **C** | Add `HighlighterFactory` fallback chain | Tree-sitter → regex fallback for unsupported languages |
| **D** | Bundle top-6 grammars | Python, JS/TS, Rust, Go, JSON/YAML highlighted |
| **E** | Background thread parsing for large files | No UI stutter on files >10K lines |
| **F** | LSP semantic token overlay | Full semantic highlighting when server is active |

## Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| Grammar C files increase binary size | ~200KB per language; 20 languages ≈ 4MB — acceptable for a native IDE |
| Tree-sitter API changes | Pin to a tagged release via FetchContent GIT_TAG |
| Worker thread parse race | Use `TSTree` copy-on-write (tree-sitter trees are immutable after creation) |
| Regex highlighter diverges during transition | Keep both paths; factory selects tree-sitter when grammar is available, regex otherwise |
| LLVM MinGW compatibility | Tree-sitter is pure C99 — compiles with any toolchain |

## Prototype Acceptance Criteria

The research phase is considered complete when:

1. Tree-sitter core library compiles as part of the Exorcist CMake build
2. C++ grammar loaded and produces a parse tree for a sample file
3. `TreeSitterHighlighter` correctly colours keywords, strings, comments, types, and functions in a C++ file
4. Output quality compared side-by-side with the existing regex highlighter shows improvement in at least: nested comments, template expressions, and raw string literals
5. No regression in startup time (tree-sitter init < 5ms for a single grammar)
